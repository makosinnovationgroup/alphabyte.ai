## The pattern we see every time {#the-pattern-we-see-every-time}

A company decides to deploy AI. They pick a use case, hire a vendor or an internal team, and start building. Four months later, the model works — in the demo environment, with clean sample data. Then they connect it to production data and everything falls apart.

The customer records have three different formats for phone numbers. The product catalogue has 12,000 duplicate entries. The financial data is spread across four systems that disagree on what "revenue" means. The addresses are inconsistent. The dates are ambiguous. The categories are whatever the last person who touched the spreadsheet decided they should be.

This is not an edge case. This is the norm. Over half of the AI projects that fail in mid-market organizations fail because of data quality, not model quality.

We refuse to build before we validate. Not because we are cautious. Because we have seen what happens when you skip this step.

## What data readiness actually means {#what-data-readiness-actually-means}

Data readiness is not a vague assessment. It is a specific set of conditions that must be true before an AI system can operate reliably against a data source.

The data must be accessible — not locked in a system that requires a six-week IT project to extract from. The data must be consistent — the same field must mean the same thing across records. The data must be current — stale data produces stale results. And the data must be structured well enough that a model can reason over it without constant disambiguation.

> "Over half of the AI projects that fail in mid-market organizations fail because of data quality, not model quality."

None of these conditions are exotic. They are basic data hygiene. But in most mid-market organizations, they are not met — because nobody has had a reason to enforce them until now.

## The first-week audit {#the-first-week-audit}

In the first week of every engagement, before we write a line of application code, we run a data audit against the systems the AI will need to access.

We look at five things. Schema consistency — are the tables and fields structured the way the documentation says they are? Value quality — are the actual values clean, complete, and unambiguous? Freshness — when was the data last updated, and is the update cadence sufficient? Accessibility — can we connect to the system programmatically, with appropriate permissions? And volume — is there enough data for the use case, or are we working with a sample that will not generalize?

The audit produces a clear report. Green, yellow, or red on each dimension. Green means we can build against it. Yellow means we need to clean or restructure before building. Red means the data source is not viable for the use case without significant remediation.

This takes three to five days. It saves months.

## When the data is not ready {#when-the-data-is-not-ready}

Sometimes the audit comes back red. The data is too messy, too fragmented, or too inaccessible to support the planned AI use case.

When that happens, we say so. We do not build a demo on clean sample data and hope the production issues resolve themselves. We scope a data readiness engagement — a focused project to clean, restructure, and connect the data sources that the AI system needs.

This is not glamorous work. It is deduplication, schema normalization, pipeline construction, and governance. But it is the work that makes everything after it possible.

The organizations that treat data readiness as a prerequisite — not an afterthought — are the ones whose AI systems actually make it to production. Every time.
