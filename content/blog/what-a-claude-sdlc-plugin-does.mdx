## The governance gap {#the-governance-gap}

Your team is already using Claude. Some of them are using it well. Some are pasting sensitive data into personal accounts. Some are building workflows that depend on prompt patterns nobody else understands.

This is the governance gap. AI usage is happening. It is producing value. But it is ungoverned, undocumented, and unreplicable. If the person who built the workflow leaves, the workflow leaves with them.

Most organizations try to close this gap with policy. Write an acceptable-use document. Send an email. Hope people follow it.

Policy without tooling is a suggestion. The SDLC plugin is the tooling.

## What the plugin does {#what-the-plugin-does}

The Claude SDLC plugin sits between your team and Claude. It does not restrict what people can build. It does enforce how they build it.

Every project created through the plugin has a defined scope — which data sources it can access, which operations it can perform, and what the output looks like. Every conversation is logged. Every tool call is auditable. Every prompt pattern is versioned.

When a team member builds something useful, the plugin captures it as a replicable pattern. The next person who wants to build something similar starts from a working template, not a blank page.

> "Policy without tooling is a suggestion. The SDLC plugin is the tooling."

The plugin also enforces a graduation path. A workflow starts as a personal tool. When it proves useful, it can be promoted to a team tool — reviewed, tested, and available to others. When it reaches production maturity, it graduates to an organizational asset with monitoring and support.

## What your team sees {#what-your-team-sees}

From the user's perspective, the plugin is lightweight. They open Claude. They select or create a project. They build.

The difference is that the environment is configured. Data connections are pre-wired through MCP. Permissions are set. Guardrails are active. The user does not need to think about governance — it is embedded in the environment.

When they build something, it is automatically documented. The prompt patterns, the data sources, the tool calls — all captured. If someone else wants to understand or extend the workflow, the documentation is already there.

When they make a mistake — and they will — the plugin catches it. Sensitive data in an output gets flagged. An operation outside the permitted scope gets blocked. A prompt pattern that produces inconsistent results gets surfaced for review.

## Why this is the first thing we deploy {#why-this-is-the-first-thing-we-deploy}

We deploy the SDLC plugin before we build any applications. Before we automate any workflows. Before we connect any data sources.

Because everything that comes after depends on it. The citizen development programme needs governance infrastructure. The MCP connections need permission boundaries. The applications your team builds need a path from experiment to production.

The plugin is that infrastructure. It takes two days to configure and deploy. By day three, your team is building inside a governed environment with real data connections and a clear graduation path.

Every application built from that point forward is auditable, replicable, and maintainable. That is the difference between informal AI usage and an organizational capability.
